# ChapterGraph Research Report

## Scope

This report is based on a full pass through the repository source, generated artifacts, and local helper scripts in `c:\Users\hy\ChapterGraph`.

I read the project as three connected systems:

1. A raw-book parsing and enrichment pipeline in `feature_achievement/`.
2. A PostgreSQL-backed FastAPI service that stores and serves graph data.
3. A lightweight canvas frontend in `frontend/` that visualizes books and chapters.

I also inspected ignored/generated folders that still affect understanding, especially `output/`, `frontend/graph-core-dist/`, and `backend/`.

## Executive Summary

ChapterGraph builds a chapter-level cross-book knowledge graph from technical book tables of contents. The project does not parse full chapter prose. It parses structured TOC-like text files, derives a compact `chapter_text` from subsection bullets or section names, computes cross-book similarity edges, stores the result in PostgreSQL, and exposes a `/graph` API for a browser-based graph viewer.

The current tracked source of truth is `feature_achievement/`, not `backend/`. The `backend/` directory appears to be an ignored bytecode mirror containing only `.pyc` files.

The retrieval logic has two architectural layers:

- An older prototype flow in `feature_achievement/edge_generation2.py` and `feature_achievement/pipeline.py`.
- A newer pluggable retrieval architecture under `feature_achievement/retrieval/` that is what the API actually uses.

The current practical pipeline is:

1. Load cached enriched JSON from `output/` if present.
2. Otherwise parse raw TOC text from `book_content/`.
3. Build `chapter_text` strings from bullet/section signals.
4. Build TF-IDF features across chapters.
5. Generate candidates by overlapping top TF-IDF tokens.
6. Score candidates with TF-IDF cosine similarity or optional embeddings.
7. Persist books, chapters, runs, and edges to PostgreSQL.
8. Return the graph to a React + D3 canvas frontend.

## End-to-End Runtime Flow

### 1. Raw input format

The parser expects text files that look like TOCs:

- chapter lines
- section lines
- bullet/subsection lines
- optional page numbers at line ends

The parsing is heuristic and format-sensitive. It is not a generic document parser.

Examples:

- `1 Getting started with Spring 3`
- `1.1 What is Spring? 4`
- `Initializing a Spring project with Spring Tool Suite 7`
- `Chapter 1 An Introduction to Java`
- `1.2.1 Simple`

### 2. Parsing into structured chapter data

`feature_achievement/ingestion.py` converts each source text file into a structure like:

```json
{
  "book_id": "...",
  "parser_meta": {
    "chapter_types": ["int" or "Chapter"],
    "section_types": ["has_page", "no_page"],
    "bullet_types": ["mix", "single"],
    "rule": "default_v1"
  },
  "chapters": [
    {
      "id": "book-id::ch1",
      "order": 1,
      "title": "...",
      "sections": [...],
      "signals": {
        "bullets": [...],
        "raw_text": ""
      }
    }
  ]
}
```

Important parser behavior:

- A line is treated as a chapter if the first token is numeric or exactly `Chapter`.
- A line is treated as a section if the first token contains exactly one dot.
- Everything else becomes a bullet line.
- Chapter IDs are normalized as `book_id::ch{number}`.
- Section titles drop numeric prefixes and usually drop trailing page numbers.
- Bullet text is collected into `signals.bullets`.

### 3. Enrichment

`feature_achievement/enrichment.py` is intentionally simple right now. The only active enrichment is `enrich_chapter_text()`:

- If a chapter has bullets, `chapter_text` becomes all bullets joined together.
- Otherwise, `chapter_text` becomes all section titles joined together.

This means retrieval is based on TOC signals, not on full content.

The module caches enriched results to `output/{book_id}_enriched.json`, and `load_all_enriched_data()` uses `prefer_output=True` by default, so local runs usually read cached JSON instead of reparsing source text.

### 4. Retrieval

The API uses the pluggable retrieval system under `feature_achievement/retrieval/`.

Shared resources:

- `collect_chapter_texts()` produces `dict[chapter_id] -> chapter_text`.
- `build_tfidf_index()` builds a corpus-wide TF-IDF matrix with:
  - `ngram_range=(1, 2)`
  - English stop words
  - `min_df=1`

Candidate generation:

- `extract_top_tfidf_tokens()` keeps each chapter's top 20 tokens by weight.
- `build_token_index()` maps token -> set of chapter IDs.
- `TfidfTokenCandidateGenerator.generate()` counts shared top tokens.
- Only candidates with at least 2 shared top tokens survive.

Scoring:

- `TfidfSimilarityScorer` uses cosine similarity over the sparse TF-IDF matrix.
- `EmbeddingSimilarityScorer` uses normalized sentence-transformer embeddings and scores by dot product.

Pipeline logic:

- `RetrievalPipeline.retrieve()` filters out intra-book matches.
- Only scores at or above `min_score` are emitted.
- Edge shape is:

```json
{
  "from": "source chapter id",
  "to": "target chapter id",
  "score": 0.42,
  "type": "tfidf" or "embedding"
}
```

Edge generation is directional in storage. For symmetrical pairs, the system typically stores both directions because each chapter is queried independently.

### 5. Persistence

The DB layer is in `feature_achievement/db/`.

Tables:

- `Book`
- `Chapter`
- `EnrichedChapter`
- `Edge`
- `Run`

Typical `/compute-edges` flow:

1. Insert a `Run` row first.
2. Load the selected books' enriched chapter data.
3. Build retrieval resources.
4. Generate edges.
5. Persist `Book` and `Chapter` rows if missing.
6. Persist `Edge` rows tied to `run_id`.

`Run.book_ids` is stored as a JSON string, not as a relational join table.

### 6. API serving

The FastAPI app is in `feature_achievement/api/main.py` and mounts a single router from `feature_achievement/api/routers/edges.py`.

Primary endpoints:

- `POST /compute-edges`
- `GET /edges?book_id=...`
- `GET /graph?run_id=...`
- `GET /runs`

`GET /graph` returns a frontend-oriented shape:

- book nodes
- chapter nodes
- edge list for one run

### 7. Frontend rendering

The frontend is deliberately thin:

- `frontend/index.html` loads D3 from CDN and `app.js` as an ES module.
- `frontend/app.js` loads React and ReactDOM from `esm.sh`.
- `frontend/graph-core/*.ts` contains the graph state types, reducer, and D3/canvas view logic.
- `npm run build:core` compiles `graph-core/*.ts` into `graph-core-dist/`, which `app.js` imports.

The UI fetches runs, defaults to the newest one, fetches `/graph?run_id=...`, and draws a force-directed view on a canvas.

Interaction model:

- scroll/pinch/drag handled by D3 zoom
- hover shows a tooltip
- double-click toggles a book between collapsed and expanded

When collapsed:

- a book is one node
- the node shows chapter count at its center

When expanded:

- the book node disappears
- chapter nodes for that book appear
- only chapter-to-chapter edges whose endpoints are both visible are drawn

## Detailed Module Notes

### `feature_achievement/ingestion.py`

This file is the parser core and defines the project's most important assumptions.

Key specifics:

- `detect_role()` uses extremely simple token-shape heuristics.
- `detect_chapter_type()` supports two chapter formats:
  - leading integer
  - leading `Chapter` keyword
- `detect_section_type()` only checks whether the last token is numeric.
- `detect_bullet_type()` has two modes:
  - `mix`: text line without numeric bullet prefix
  - `single`: token like `1.2.3`

The parser is tolerant of mixed TOC formats, but only within a narrow family of numbering conventions.

One subtle behavior: `current_bullet` is accumulated across lines. That means a bullet can span multiple physical lines until the parser sees a numeric token that triggers a flush.

The field `signals.raw_text` exists in the chapter structure but is never populated anywhere in the current code.

### `feature_achievement/enrichment.py`

The only active enrichment is chapter-text creation. No keyphrase extraction or NLP-based enrichment is currently active.

Important specificity:

- The file imports spaCy and loads `en_core_web_sm` at module import time.
- The loaded `nlp` object is not used by the current enrichment code.

That means this module imposes an environment dependency that does not currently buy any behavior.

### `feature_achievement/edge_generation2.py`

This is the pre-refactor retrieval implementation. It still works as a standalone algorithm, but it duplicates logic now found in `feature_achievement/retrieval/`.

It is useful as history/reference, but it is not the active API path.

### `feature_achievement/retrieval/`

This is the cleaner architectural layer.

The design separates:

- candidate generation
- similarity scoring
- orchestration

That abstraction is real, but in the current API only one candidate generator is actually used: TF-IDF top-token overlap.

### `feature_achievement/db/models.py`

The data model reflects two different storage strategies:

- `Chapter`: minimal chapter data used by the graph API.
- `EnrichedChapter`: richer imported chapter records with sections/signals stored as JSON.

This suggests the project evolved from a simple chapter-edge graph to a richer enrichment store, but the public graph API still reads from `Chapter`, not `EnrichedChapter`.

### `feature_achievement/db/crud.py`

There are two different persistence paths:

- `persist_books_and_chapters()`: used by `/compute-edges`
- `persist_enriched_chapters()`: used by `scripts/import_enriched_chapters.py`

The first path stores only a minimal chapter representation. The second stores the richer enrichment payload.

### `feature_achievement/api/routers/edges.py`

This file is the operational center of the current app.

Important specifics:

- It does not use the cached dependency helpers in `api/deps.py`; it rebuilds the retrieval resources inside the route handler.
- It loads all configured books, then filters them by requested `book_ids`.
- Candidate generation is always TF-IDF token based, regardless of request payload.
- Similarity can switch between TF-IDF and embeddings.

### `frontend/app.js`

The UI uses React for lifecycle/state shell and D3 for zoom/simulation.

Important specifics:

- It keeps canonical state in React with `useReducer`, but the force-graph rebuild path mutates `state.nodes` and `state.links` imperatively inside `rebuildGraph()`.
- Rendering is an animation loop that repaints the canvas continuously with `requestAnimationFrame`.
- Tooltips are managed by directly editing a DOM element, not via React render state.

### `frontend/graph-core/buildView.ts`

This file contains the visualization semantics.

Important specifics:

- Expanded books are replaced by their chapters; the original book node is not shown simultaneously.
- Edge visibility depends on endpoint visibility, so edges vanish unless both books are expanded.
- Chapter nodes are seeded around either the previous book position or the previous chapter centroid.
- Double-click works on any hit node because both book nodes and chapter nodes carry `bookId`.

That last point means double-clicking a chapter toggles the whole parent book, not just double-clicking the collapsed book node.

## Operational Notes

### Backend startup

The intended local startup path is:

- initialize DB with `python -m feature_achievement.scripts.init_db`
- run API with `uvicorn feature_achievement.api.main:app --reload`

`scripts/run_local.ps1` automates:

- starting FastAPI in a new PowerShell window
- building frontend graph-core
- starting a static HTTP server for the frontend

### Frontend build assumptions

The frontend depends on compiled files in `frontend/graph-core-dist/`.

If `graph-core/*.ts` changes, `npm run build:core` must be rerun before the browser app reflects those changes.

### Database dependency

The database URL is hard-coded in `feature_achievement/db/engine.py` as:

`postgresql+psycopg2://postgres:1234@localhost:5432/chaptergraph`

This repo currently assumes:

- local PostgreSQL
- fixed credentials
- no environment-based configuration layer

## Specific Findings, Risks, and Quirks

### 1. `backend/` is not real source

The `backend/` directory contains only `.pyc` files and mirrors the tracked Python package layout. It looks like a local execution artifact, not a maintained source tree. Anyone new to the repo could easily mistake it for the actual backend.

### 2. There are effectively two retrieval generations in the repo

The old procedural retrieval path still exists beside the newer modular retrieval layer. The active API path uses the new layer, but the legacy files are still present and runnable, which increases ambiguity when reading the codebase.

### 3. `ComputeEdgesRequest.similarity` default is misleading

`feature_achievement/api/routers/compute_edges_request.py` sets:

`similarity: str = "tfidf/embedding"`

But the route logic treats only `"embedding"` as embeddings and everything else as TF-IDF. So the default value is not a true enum-like default; it is a descriptive string that falls through to TF-IDF behavior.

### 4. `candidate_generator` is exposed in the API but not really configurable

The request model accepts `candidate_generator`, but `POST /compute-edges` always builds `TfidfTokenCandidateGenerator`. So this field is currently aspirational rather than functional.

### 5. `Run.min_store` is probably a typo for `min_score`

The `Run` model uses `min_store`, and the route writes `req.min_score` into that field. The code works mechanically, but the naming mismatch is a persistent semantic footgun.

### 6. `spaCy` is a required import-time dependency without current use

`feature_achievement/enrichment.py` imports spaCy and loads `en_core_web_sm` immediately, even though none of the active enrichment logic uses it. This can make startup fail on machines that do not have the model installed.

### 7. The API graph does not use `EnrichedChapter`

There is a richer `EnrichedChapter` table and a dedicated import script, but the frontend-facing graph is built from `Book`, `Chapter`, and `Edge`. That means the richer enrichment persistence path is currently orthogonal to the visible product.

### 8. `persist_books_and_chapters()` is insert-only in practice

It only adds missing `Book` and `Chapter` rows. If cached enriched JSON changes chapter titles or chapter text for existing IDs, those rows are not updated through the `/compute-edges` path.

### 9. The parser depends on TOC conventions, not semantics

The parser only understands a narrow line-shape grammar. It will work well on book TOCs similar to the existing inputs and degrade quickly on source files that do not match those conventions.

### 10. Visualization only shows inter-chapter edges when both endpoint books are expanded

This is a deliberate UX choice, but it means the collapsed overview hides all actual relationship lines. The graph is conceptually present before expansion, but visually absent.

### 11. The frontend is partly declarative and partly imperative

React owns high-level state and mounting, but D3 simulation state and canvas drawing are imperative. That is fine for this app, but it means future refactors need to respect the boundary carefully.

### 12. No test suite is present

I did not find any automated tests in the repository. The main verification model appears to be manual local runs plus ad hoc scripts.

### 13. Some terminal output shows encoding artifacts

A few comments and some README rendering appeared as mojibake when read from the terminal. That may be file-encoding drift or a console code-page issue, but it is worth noting because it affects maintainability and onboarding.

## Final Assessment

This is a compact research/prototype codebase that already has a coherent full stack:

- structured book ingestion
- enrichment caching
- pluggable retrieval
- relational persistence
- API serving
- graph visualization

Its strongest characteristic is that the pipeline is understandable end to end and can be traced quickly from raw text file to rendered graph.

Its weakest characteristic is not algorithmic complexity, but operational ambiguity:

- duplicate generations of code
- generated artifacts living beside source
- partially wired richer data model
- hard-coded environment assumptions
- request fields that imply configurability the implementation does not yet honor

In short: ChapterGraph is currently a TOC-driven chapter similarity explorer with a real full-stack implementation, sparse default graph output, and several signs of active iteration where architecture is ahead of final cleanup.
